// how it works

Every request in 4 steps

Each call passes through a pipeline before touching OpenAI — most never reach the upstream at all.

🔑 01 — FINGERPRINT

Namespace the cache

SHA-256 hash the system prompt, combine with tenant ID. This isolates one user's cache from another's entirely.

🔢 02 — EMBED

Vectorize the query

Pass the user turn through text-embedding-3-small to get a 1536-dimensional semantic vector.

🔍 03 — SEARCH

ANN lookup in Qdrant

Find the nearest cached vector. If cosine similarity clears the per-query-type threshold, serve from cache. Done.

💾 04 — STORE

Cache and return

On a miss: forward to OpenAI, store the response in Qdrant with a TTL, return to client. Next caller gets a hit.

// features

Built for production

Not a toy cache — the hard engineering problems that make this real.

🎯

Learned Threshold Calibration

A logistic regression classifier trained on (query_A, query_B, should_cache) pairs replaces fixed cosine thresholds. ~15% better recall, no increase in false positives.

🏢

Multi-Tenancy

Each tenant namespaced by {tenant_id}:{system_fingerprint}. Pass X-Tenant-ID — one tenant's responses never contaminate another's.

📡

Streaming SSE

Buffers streamed tokens, writes to cache, re-streams from cache on hits. Works with "stream": true — the client never knows the difference.

⏱️

TTL Invalidation

Every entry carries a configurable TTL. Expired entries evicted lazily on lookup. Per-namespace invalidation API for topic-drift handling.

🚀

Cold-Start Pre-Warming

Historical query logs clustered via k-means. The most representative query per centroid is pre-generated and seeded — solving the empty-cache launch problem.

🔌

Drop-in Compatible

Fully OpenAI-compatible. Change base_url, nothing else. Works with any SDK or client targeting /v1/chat/completions.
